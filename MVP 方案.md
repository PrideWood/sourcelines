# **Quotes 项目 MVP 方案（正式草案）**







## **一、MVP 产品目标**







### **1.1 产品目标**





在第一阶段，Quotes 不追求做成开放式大社区，而是先验证下面三件事：



- 用户是否愿意浏览“有出处、有原文、有分类”的短文本内容
- 用户是否愿意提交符合规范的文本投稿
- 平台是否能够以较低成本完成基础审核、分类与检索







### **1.2 MVP 核心定位**





MVP 应定位为：



**一个可审核、可检索、可收藏的多语言原文短文本资料平台**



它不是：



- 泛社交平台
- 名人名言搬运站
- 自动翻译句子站
- 纯推荐流产品





它优先解决的是：



- 原文真实性
- 出处可验证性
- 分类浏览体验
- 学习使用价值







### **1.3 MVP 成功标准**





首阶段不以 DAU 为唯一指标，建议关注：



- 投稿数
- 投稿通过率
- 有完整出处证据的投稿占比
- 内容收藏率
- 搜索使用率
- 标签页、作品页访问占比
- 用户二次回访率

## **2.1 用户端功能**







### **A. 浏览与发现**







#### **A1. 首页**



功能：



- 今日精选句子
- 最新通过审核内容
- 按语言入口
- 按体裁入口
- 按主题入口
- 按难度入口
- 搜索框





目的：



- 帮助用户快速理解平台内容结构
- 不依赖算法流，先以编辑式和分类式导航为主







#### **A2. 内容列表页**



支持维度：



- 语言
- 体裁
- 主题
- 作品
- 难度
- 标签
- 最新 / 热门 / 精选





筛选项建议：



- 语言
- 是否含译文
- 是否证据完整
- 难度等级
- 来源类型（诗歌 / 小说 / 电影 / 演讲等）





排序方式：



- 最新收录
- 最多收藏
- 编辑精选







#### **A3. 搜索功能**



搜索范围：



- 原文文本
- 作者名
- 作品名
- 标签
- 角色名
- 语言





MVP 阶段建议：



- 先做基础全文搜索 + 结构化筛选
- 不做复杂语义搜索

#### **B1. 句子详情页**



展示字段：



- 原文

交互功能：



- 切换原文 / 原文+译文
- 展开上下文
- 跳转到作品页 / 作者页 / 标签页
- 收藏
- 分享链接







#### **B2. 作品页**



展示内容：



- 作品名
- 原语言
- 作者 / 导演 / 编剧
- 体裁
- 简介
- 相关句子列表
- 相关标签
- 相关语言信息







#### **B3. 作者页**



展示内容：



- 作者名
- 语言 / 国籍（可选）
- 简介

- 相关句子

- 邮箱登录

#### **C2. 收藏功能**

- 可选译文

- 待审核

用户可查看：

### **D. 审核系统**

后台应支持查看：



- 投稿原文
- 语言
- 出处信息
- 证据
- 投稿人
- 系统初筛结果

操作：



- 通过
- 退回补充

后台可管理：



- 标签创建
- 标签层级
- 标签别名
- 标签合并

管理员可修改：



- 文本格式

- 难度

- 作品绑定
- 作者绑定







#### **D4. 作品管理**



管理员可创建 / 编辑：



- 作品信息

- 作者关联

#### **D5. 作者管理**



管理员可创建 / 编辑：



- 作者姓名
- 别名

- 作品关联

## **2.3 系统辅助功能**







### **E. 自动化辅助（MVP 只做基础版）**







#### **E1. 基础重复检测**



检测维度：



- 完全重复原文
- 同作品内高相似文本
- 同作者高相似文本







#### **E2. 结构完整性检查**



检查是否缺失：



- 语言
- 原文
- 作品名
- 出处位置
- 证据







#### **E3. 敏感内容检查**



自动标记：



- 明显违规内容
- 可能侵权的大段文本
- 低质量垃圾投稿





------





## **2.4 MVP 不做的功能**





为了控制范围，首版建议明确不做：



- 私信
- 评论区大战
- 用户互相关注
- AI 自动翻译发布
- AI 自动生成伪“赏析”
- 音频朗读
- 背诵练习模式
- Anki 导出
- App 原生端
- 复杂推荐算法
- 商业化会员体系





这些都可以在内容质量和检索体验稳定后再做。

# **九、技术实现上的实际建议**





如果你想走比较稳的 Web MVP 路线，我建议技术上优先考虑这种组合：



- 前端：Next.js
- UI：Tailwind CSS + shadcn/ui
- 后端：Next.js API 或独立 Node 服务
- 数据库：PostgreSQL
- ORM：Prisma
- 文件存储：Supabase Storage / Cloudflare R2 / S3 兼容存储
- 搜索：先用 PostgreSQL 全文搜索，后续再考虑 Meilisearch





原因很简单：



- 开发快
- 结构清楚
- 管理后台容易搭
- 以后扩展 App 或 API 也方便

# **十、你接下来最应该先定下来的 5 件事**





在正式开工前，我建议你优先把下面五件事写死：





## **1. 投稿最小字段**





哪些字段是强制的，哪些是可选的。





## **2. 证据合格标准**





什么算“可验证”，什么不算。





## **3. 难度分级规则**





是人工分级，还是平台规则辅助分级。





## **4. 标签治理规则**





谁能提标签，谁能建标签，谁能改标签。





## **5. 版权边界**





每条文本最长允许多少字，哪些来源必须谨慎展示。
